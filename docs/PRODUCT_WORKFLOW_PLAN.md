# Детальный план развития e-commerce flow

## 1) Целевой сценарий пользователя

1. Пользователь открывает сайт и видит каталог.
2. Каталог и фильтры формируются из данных, которые завёл администратор (категории, размеры, цвета, цены, доступность).
3. Пользователь добавляет товары в корзину и меняет количество в пределах доступного остатка.
4. Если пользователь не авторизован — корзина живёт локально (cookie/localStorage).
5. После логина/регистрации гостевая корзина объединяется с серверной корзиной.
6. Перед оформлением проводится валидация корзины (остатки, актуальность цен, статус товара).
7. Пользователь подтверждает обязательные контакты (email/телефон — по настройке в админке).
8. Пользователь вводит адрес (с подсказками DaData), доставка считается через Яндекс.Доставку.
9. Пользователь видит итог: товары + доставка + итог к оплате + выбор метода оплаты.
10. После оплаты создаётся история заказа у пользователя и обновляется статус у администратора.

## 2) Что обязательно должно быть в админке

### 2.1 Товары и справочники
- CRUD товаров.
- Варианты товара (SKU): размер, цвет, артикул, вес/габариты.
- Остатки по SKU (складской учёт).
- Категории/подкатегории/бренды.
- Атрибуты для фильтров (материал, сезон и т.д.).
- Медиа: фото/видео.

### 2.2 Настройки checkout
- Флаги обязательности контактов:
  - требовать email;
  - требовать телефон;
  - требовать оба.
- Время резерва товара в корзине (например, 15–30 минут) — опционально.
- Политика обработки изменений цены:
  - автообновлять цену в корзине;
  - требовать подтверждения пользователя.

### 2.3 Заказы и статусы
- Статусы оплаты: `pending`, `paid`, `failed`, `refunded`.
- Статусы доставки: `new`, `processing`, `shipped`, `delivered`, `cancelled`.
- Лог событий заказа (кто и когда менял статус).
- Экран «новые заказы» + уведомления.

## 3) Корзина: логика для гостя и авторизованного пользователя

### 3.1 Хранение корзины
- Гость: `localStorage` + `guestCartId` (UUID).
- Авторизованный: `cart_items` в БД.

### 3.2 Объединение при логине
Алгоритм merge:
1. Берём гостевые позиции.
2. Для каждой позиции ищем пару `productId + sku + size + color` в серверной корзине.
3. Суммируем количество, но не выше доступного остатка.
4. Конфликтные позиции помечаем сообщениями:
   - «остаток уменьшен»;
   - «товар недоступен»;
   - «цена изменилась».
5. Сохраняем snapshot результата и показываем пользователю итог merge.

### 3.3 Валидация корзины
Проверка при каждом открытии корзины и перед созданием заказа:
- SKU существует;
- товар активен;
- цена не изменилась (или отмечено изменение);
- остаток >= запрошенного количества.

## 4) Заказ и оплата

### 4.1 Checkout шаги
1. Контакты.
2. Адрес + подсказки DaData.
3. Выбор доставки + расчёт стоимости (Яндекс.Доставка).
4. Подтверждение состава заказа и итоговой цены.
5. Выбор способа оплаты.

### 4.2 Способы оплаты
- Перевод по номеру телефона.
- Перевод на расчётный счёт.
- СБП.
- ЮMoney.
- Сбербанк (эквайринг/инвойс).

Рекомендуется единый интерфейс платёжного провайдера через `PaymentService` (адаптеры), чтобы не зашивать бизнес-логику под один способ.

### 4.3 Надёжность платежей
- Idempotency key на создание платежа.
- Webhook endpoint для подтверждения оплаты.
- Повторная обработка webhook (без дублей).
- Сверка суммы платежа и суммы заказа.

## 5) Интеграции

### 5.1 DaData
- Endpoint автокомплита адресов.
- Нормализация адреса и хранение `fias/geo`-идентификаторов.
- Кеш подсказок (короткоживущий) для снижения стоимости запросов.

### 5.2 Яндекс.Доставка
- Расчёт тарифа по адресу, весу, габаритам, SLA.
- Сохранение выбранного тарифа в заказе snapshot-ом.
- Повторный расчёт, если изменился адрес/состав корзины.

## 6) Модель данных (минимум)

- `products`
- `product_variants` (size/color/sku)
- `inventory`
- `categories`
- `attributes`, `attribute_values`
- `cart_items`
- `orders`
- `order_items`
- `order_events`
- `payments`
- `shipments`
- `app_settings`

Важно: в `order_items` хранить snapshot названия/цены/атрибутов на момент заказа, чтобы история не ломалась после редактирования товара.

## 7) API-контракты (уровень endpoint-ов)

- `POST /api/cart/items` — добавить/обновить позицию.
- `POST /api/cart/merge` — merge гостевой корзины.
- `POST /api/checkout/validate` — финальная валидация.
- `POST /api/checkout/shipping-quote` — расчёт доставки.
- `POST /api/orders` — создать заказ.
- `POST /api/payments` — создать платёж.
- `POST /api/payments/webhook` — webhook провайдера.
- `GET /api/profile/orders` — история заказов пользователя.
- `PATCH /api/admin/orders/{id}/status` — изменение статуса админом.

## 8) Этапы внедрения

### Этап 1 (быстрый запуск)
- Стабилизировать запуск backend (локальный SQLite + PostgreSQL в проде).
- Вынести настройки checkout в `app_settings`.
- Добавить `product_variants` и ограничения остатков.

### Этап 2 (корзина и checkout)
- Merge гостевой корзины.
- Валидация корзины с маркерами проблем.
- Шаги checkout + обязательные поля по настройкам.

### Этап 3 (оплата и доставка)
- DaData autocomplete.
- Яндекс.Доставка quote.
- Payment adapters + webhook + статусы.

### Этап 4 (операционка)
- Админ-уведомления.
- Полная история событий заказа.
- Отчёты и аналитика.

## 9) Почему сейчас «приложение не запускается» и как исправлено

Типичный локальный сбой: backend ожидает PostgreSQL по умолчанию, и если БД не поднята — приложение завершается на старте.

Решение в этом репозитории:
- backend теперь автоматически использует SQLite (`backend/app.db`) если не задан `DATABASE_URL`;
- PostgreSQL включается автоматически, когда `DATABASE_URL` передан явно;
- добавлен явный лог активного провайдера БД при старте.

Это позволяет разработчику запускать API «из коробки» без обязательного локального PostgreSQL.
